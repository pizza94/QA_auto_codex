# 테스트 계획서

이 문서는 IRUDA QA 프로젝트의 Playwright 자동화 테스트 시나리오를 관리한다.

실제 비밀번호, 세션 쿠키, 일회용 토큰, 채팅 프롬프트 원문은 이 문서에 저장하지 않는다. 테스트 목적, 기대 결과, 필요한 환경값, 검증 이력만 기록한다.

## 1. 업데이트 규칙

테스트 시나리오 구현이 완료되고 최종 변경사항이 GitHub에 푸시될 때마다 이 문서를 함께 업데이트한다.

각 업데이트에는 다음 내용을 포함한다.

1. 시나리오 번호와 이름
2. 테스트 파일 경로
3. 테스트 목적과 기대 결과
4. 주요 셀렉터 또는 검증 기준
5. 마지막 검증 실행 결과
6. CI 제한사항 또는 후속 작업

새 시나리오는 `TC-001`, `TC-002`, `TC-003` 순서로 번호를 부여한다. 목차의 순서도 현재 시나리오 목록과 동일하게 유지한다.

## 2. 환경 정보

- 애플리케이션 기본 URL: `PLAYWRIGHT_BASE_URL` 환경변수로 설정
- 로그인 사용자 ID: `PLAYWRIGHT_USERNAME` 환경변수로 설정
- 로그인 비밀번호: `PLAYWRIGHT_PASSWORD` 환경변수로 설정
- 로컬 대상 URL 예시: `http://10.194.5.53:8180`

## 3. 현재 시나리오

### TC-001 로그인

상태: 구현 완료, 로컬 검증 완료

파일: `tests/login.spec.ts`

목적:

1. IRUDA 로그인 페이지를 연다.
2. 환경변수에서 받은 사용자 ID와 비밀번호를 입력한다.
3. 로그인 폼을 제출한다.
4. 브라우저가 로그인 URL에서 벗어났는지 확인한다.
5. Data Portal 메인 화면이 표시되는지 확인한다.

주요 셀렉터:

1. 사용자 ID 입력칸: `#usrId`
2. 비밀번호 입력칸: `#pswd`
3. 로그인 버튼: `button.submit`
4. 로그인 성공 확인 기준: `Welcome to Data Portal` 제목 표시

기대 결과:

1. 로그인이 15초 이내에 완료된다.
2. 최종 URL이 `/iruda_woori/login`이 아니다.
3. Data Portal 제목이 화면에 표시된다.

마지막 검증 결과:

- 환경변수 계정을 사용한 로컬 Chromium 실행: 통과
- GitHub Actions 실행: 계정 정보가 없어 로그인 테스트는 스킵, smoke 테스트는 통과

### TC-002 QualityStream 메뉴 진입 및 구조 확인

상태: 구현 완료, 로컬 검증 완료

파일: `tests/qualitystream.spec.ts`

참고 문서: `docs/qualitystream-menu-map.md`

목적:

1. 로그인 후 Data Portal 메인 화면에서 `품질관리` 카드를 클릭한다.
2. QualityStream 대시보드로 정상 이동했는지 확인한다.
3. QualityStream 상위 메뉴가 실제 화면 기준으로 순서대로 표시되는지 확인한다.
4. 로컬 TC Excel 기능 시트와 대응되는 주요 하위 메뉴 링크가 존재하는지 확인한다.

주요 셀렉터 및 검증 기준:

1. 품질관리 카드: `.page-item` 중 텍스트가 `품질관리`인 요소
2. QualityStream URL: `/quality_woori/main#dashboard`
3. 브라우저 제목: `QualityStream 4.3`
4. 상위 메뉴: `a.menu-item`
5. 하위 메뉴: 각 기능별 `href` 링크

기대 결과:

1. 품질관리 클릭 후 QualityStream 대시보드로 이동한다.
2. 상위 메뉴 12개가 표시된다.
3. TC 기능 영역을 대표하는 하위 메뉴 링크가 모두 존재한다.

마지막 검증 결과:

- 실제 QualityStream 메뉴 구조 수집 완료
- 자동화 테스트 구현 완료
- 환경변수 계정을 사용한 로컬 Chromium 실행: 통과
- 자격증명 없는 CI 유사 실행: QualityStream 테스트 스킵, smoke 테스트 통과

### TC-003 데이터품질지표(DQI)관리 CRUD 및 Search 검증

상태: 구현 완료, 브라우저 자동화 수동 검증 완료

파일: `tests/dqi-management.spec.ts`

참고 문서: `docs/dqi-management-run-2026-07-03.md`

목적:

1. `검증기준관리 > 데이터품질지표(DQI)관리` 화면에 진입한다.
2. 필수 입력 항목 라벨 우측의 빨간 별표 표시를 검증한다.
3. 필수값 미입력 시 저장되지 않고 검증 메시지가 표시되는지 확인한다.
4. 상단 `신규` 버튼으로 DQI 대분류를 생성한다.
5. 생성한 대분류 노드 우클릭 후 컨텍스트 메뉴의 `신규`로 DQI 소분류를 생성한다.
6. 트리 모두열기/모두닫기, Search 완전일치/부분일치, 수정, 삭제를 검증한다.
7. BR상세정보와 연관BR 리포트 버튼의 빈 결과 안내 동작을 검증한다.

주요 셀렉터 및 검증 기준:

1. DQI 영역: `#dqiRegion`
2. 신규 버튼: `#newNodeButton`
3. 저장 버튼: `#dqiSaveButton`
4. 트리 검색: `#dqiTree #tableTreeSearch`
5. 모두열기/모두닫기: `#allOpenButton`
6. 대분류명: `[name="dqiName"]`
7. 소분류명: `[name="subDqiName"]`
8. 대분류 설명: `[name="dqiDesc"]`
9. 소분류 설명: `[name="subDqiDesc"]`
10. 목표수준: `[name="desiredQuality"]`
11. BR상세정보: `#brDetailInfo`
12. 연관BR 리포트: `#brReport`
13. 트리 우클릭 메뉴: `.vakata-context`

기대 결과:

1. 필수 라벨의 `::after` 가상요소가 `*`이고 색상이 `rgb(217, 45, 32)`이다.
2. 필수값 미입력 저장 시 `필수 입력입니다.`가 표시된다.
3. 신규 DQI 대분류와 소분류가 저장되고 트리와 상세영역 값이 일치한다.
4. 수정 후 재진입해도 변경값이 유지된다.
5. Search 완전일치와 부분일치 입력 시 생성한 DQI가 조회된다.
6. 삭제 후 재조회 시 테스트 데이터가 더 이상 표시되지 않는다.

마지막 검증 결과:

- 브라우저 자동화로 실제 DQI 생성, 소분류 생성, 수정, Search, 삭제까지 확인: 통과
- Search 미존재값은 빈 결과가 아니라 전체 트리가 다시 보이는 동작 확인: 요구사항 확인 필요
- BR상세정보는 BR 행 미선택 상태에서 `연관 BR정보를 선택해 주세요.` 알림 표시
- 연관BR 리포트는 관련 BR 결과가 없어 `출력할 검색 결과가 없습니다.` 알림 표시
- 테스트 데이터 삭제 후 재조회에서 미노출 확인

## 5. CI 참고사항

GitHub hosted runner는 일반적으로 사설 `10.x` 네트워크 주소에 접근할 수 없다. 내부 IRUDA 테스트를 CI에서 실행하려면 다음 중 하나가 필요하다.

1. 내부망에 self-hosted runner를 구성한다.
2. runner가 VPN 또는 내부망에 접근할 수 있게 구성한다.
3. 로그인 테스트는 로컬에서 실행하고 GitHub Actions는 공개 접근 가능하거나 mock 가능한 테스트 위주로 유지한다.

계정 정보가 설정되지 않은 경우 로그인/QualityStream/DQI 테스트는 스킵되고 smoke 테스트는 계속 실행된다.
